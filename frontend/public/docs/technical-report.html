1. Abstract

The Artwork Provenance Platform (ArP) is a Semantic Web-based system designed to manage, track, and analyze the provenance of art objects. By leveraging strict ontology-based data modeling and Linked Data principles, ArP enables interoperable exchange of cultural heritage data. This technical report documents the system's architecture, internal data structures, API specifications, RDF knowledge models, and conformance to Linked Data standards, demonstrating its capability to integrate with external knowledge bases like DBpedia, Wikidata, and the Getty Art & Architecture Thesaurus.

The platform serves researchers, museum curators, and art enthusiasts by providing a rich, interconnected view of artwork histories, ownership chains, and cultural significance. Through its use of semantic technologies, ArP transforms disparate art historical data into a cohesive, queryable knowledge graph.

2. System Architecture

2.1 Overview

ArP follows a modern three-tier architecture with clear separation of concerns. The system is containerized using Docker, enabling consistent deployment across environments. Each component communicates through well-defined interfaces: REST/JSON between frontend and backend, and SPARQL between backend and the triple store.

2.2 Frontend Architecture

The frontend is a Single Page Application (SPA) built with React 18 and TypeScript. It uses React Router v6 for client-side navigation and TanStack Query for server state management with automatic caching and background refetching. The design system is built on Tailwind CSS v4 with custom theme variables for a gallery-inspired aesthetic.

Key pages include the Artwork Gallery, Artwork Detail with provenance timeline, SPARQL Explorer, Search with faceted filtering, and Statistics Dashboard. All pages implement responsive design and accessibility best practices.

2.3 Backend Architecture

The backend is organized following a service-oriented architecture:

  • Routers: FastAPI routers define HTTP endpoints, handle request validation via Pydantic, and delegate to services.
  • Services: Business logic layer that constructs SPARQL queries, processes results, and manages caching.
  • Models: Pydantic models for request/response validation and JSON-LD context injection.
  • Config: Centralized configuration via environment variables with sensible defaults.

2.4 Data Store: Apache Jena Fuseki

Fuseki serves as the primary data store, configured with TDB2 for persistent storage. It exposes both query (/query) and update (/update) endpoints. The dataset is preloaded with the ArP ontology definition and sample artwork data in Turtle format.

3. Internal Data Structures & Models

3.1 Pydantic Models Overview

The backend uses Pydantic v2 for all data validation, serialization, and JSON-LD context injection. Models are strictly typed with optional fields supporting partial updates.

Artwork Models

The ArtworkBase class defines the core properties shared across create, update, and response operations:

class ArtworkBase(BaseModel):
    title: str = Field(..., min_length=1, max_length=500)
    artist: Optional[str] = None
    dateCreated: Optional[str] = None
    medium: Optional[str] = None
    dimensions: Optional[str] = None
    description: Optional[str] = None
    imageUrl: Optional[str] = None
    currentLocation: Optional[str] = None
    period: Optional[str] = None
    style: Optional[str] = None

The ArtworkResponse extends this with additional fields for API responses:

  • id: Unique identifier (derived from RDF URI local name)
  • externalLinks: Optional links to DBpedia, Wikidata, and Getty AAT
  • @context: JSON-LD context mapping property names to RDF predicates
  • @type: JSON-LD type declaration ("Artwork")

Provenance Models

Provenance events are typed using an enumeration to ensure data consistency:

class ProvenanceEventType(str, Enum):
    CREATION = "Creation"
    SALE = "Sale"
    GIFT = "Gift"
    BEQUEST = "Bequest"
    THEFT = "Theft"
    RECOVERY = "Recovery"
    LOAN = "Loan"
    RETURN = "Return"
    ACQUISITION = "Acquisition"
    COMMISSION = "Commission"
    INHERITANCE = "Inheritance"
    TRANSFER = "Transfer"
    NATIONALIZATION = "Nationalization"

The ProvenanceRecord model captures a single event in the chain:

class ProvenanceRecord(BaseModel):
    id: str
    event: str
    date: Optional[str] = None
    owner: Optional[str] = None
    location: Optional[str] = None
    description: Optional[str] = None
    sourceUri: Optional[str] = None
    order: Optional[int] = None

Search & Faceting Models

The search functionality returns results with facet counts for filtering:

class SearchFacets(BaseModel):
    artists: List[FacetItem] = []
    periods: List[FacetItem] = []
    media: List[FacetItem] = []
    locations: List[FacetItem] = []

class SearchResponse(BaseModel):
    query: str
    total: int
    page: int
    limit: int
    results: List[Dict[str, Any]]
    facets: Optional[SearchFacets] = None

3.2 JSON-LD Context

All API responses include a JSON-LD @context that maps human-readable property names to their corresponding RDF predicates, enabling clients to interpret responses as valid Linked Data: 

JSONLD_CONTEXT = {
    "@vocab": "http://example.org/arp#",
    "dc": "http://purl.org/dc/elements/1.1/",
    "dcterms": "http://purl.org/dc/terms/",
    "schema": "http://schema.org/",
    "title": "dc:title",
    "artist": "schema:name",
    "dateCreated": "dcterms:created",
    "medium": "arp:artworkMedium",
    "imageUrl": "schema:image"
}

4. REST API Implementation

4.1 Design Principles

The API follows RESTful conventions with OpenAPI 3.1.0 specification. Key design decisions include:

  • Resource-oriented URLs: /api/artworks, /api/artworks/{id}/provenance
  • Standard HTTP methods: GET, POST, PUT, DELETE with appropriate status codes
  • Pagination: Offset-based with page and limit parameters
  • Content negotiation: JSON responses with JSON-LD context
  • Error handling: Consistent error response format with status codes 400, 404, 422, 500

4.2 Core Endpoints

Method Endpoint Description
GET /api/artworks List artworks with filters (artist, period, search)
POST /api/artworks Create new artwork
GET /api/artworks/{id} Get artwork details with external links

4.3 SPARQL Query Translation

The backend translates REST requests into SPARQL queries dynamically. For example, listing artworks with filters: 

PREFIX dc: <http://purl.org/dc/elements/1.1/>
PREFIX arp: <http://example.org/arp#>
PREFIX schema: <http://schema.org/>

SELECT DISTINCT ?artwork ?title ?artist ?period
WHERE {
    ?artwork a arp:Artwork .
    ?artwork dc:title ?title .
    FILTER(LANG(?title) = "en")
    
    OPTIONAL { ?artwork dc:creator ?artistUri . 
               ?artistUri schema:name ?artist }
    OPTIONAL { ?artwork arp:artworkPeriod ?period }
    
    # Dynamic filter injection
    FILTER(CONTAINS(LCASE(?artist), "monet"))
}
ORDER BY ?title
LIMIT 20 OFFSET 0

4.4 Update Operations

Create and update operations use SPARQL INSERT DATA and DELETE/INSERT patterns. The service layer escapes special characters and validates input before constructing queries: 

PREFIX arp: <http://example.org/arp#>

DELETE { <artwork_uri> arp:artworkMedium ?oldMedium }
INSERT { <artwork_uri> arp:artworkMedium "Oil on canvas" }
WHERE {
    <artwork_uri> a arp:Artwork .
    OPTIONAL { <artwork_uri> arp:artworkMedium ?oldMedium }
}

5. RDF-Based Knowledge Model

5.1 Ontology Design

The ArP ontology (namespace: http://example.org/arp#) extends and aligns with established vocabularies to maximize interoperability and leverage existing tooling.

Vocabulary Alignment

ArP Class Aligned With Rationale
arp:Artwork schema:VisualArtwork, crm:E22_Human-Made_Object Schema.org for web discovery, CIDOC-CRM for museum interoperability
arp:Artist schema:Person, crm:E21_Person Standard person representation
arp:ProvenanceEvent prov:Activity, crm:E8_Acquisition PROV-O for provenance tracking, CIDOC-CRM for acquisition semantics
arp:Location schema:Place, crm:E53_Place Geographic location modeling

5.2 Property Definitions

Properties are designed to capture both descriptive metadata and provenance relationships:

Object Properties (Relationships)

  • arp:hasProvenanceEvent — Links artwork to provenance events (1:N)
  • arp:fromOwner / arp:toOwner — Ownership transfer participants
  • arp:currentOwner — Current owner reference
  • arp:currentLocation — Physical location of artwork
  • owl:sameAs — Identity links to external knowledge bases

Datatype Properties

  • arp:provenanceOrder (xsd:integer) — Sequence number in provenance chain
  • arp:eventType (xsd:string) — Enumerated event type
  • arp:artworkMedium (xsd:string) — Materials and techniques
  • arp:artworkPeriod (xsd:string) — Art historical period

5.4 Expressiveness & Usage

The ontology is intentionally pragmatic rather than maximally expressive. Key design decisions:

  • Open World Assumption: Optional properties allow incomplete data
  • Multilingual Support: Language-tagged literals for titles and descriptions
  • External Linking: owl:sameAs enables data enrichment without duplication
  • Getty AAT Integration: schema:artMedium links to standardized material vocabulary

6.1 Integration Architecture

ArP integrates with three major Linked Open Data sources, each serving a distinct purpose:

Source SPARQL Endpoint Use Case

Wikidata https://query.wikidata.org/sparql High-quality images, precise dates, material data
Getty AAT https://vocab.getty.edu/sparql Standardized art vocabulary, material taxonomy

6.2 Caching Strategy

The ExternalSPARQLService implements a TTL-based in-memory cache to minimize external requests: 

class SimpleCache:
    def __init__(self, ttl: int = 3600):
        self._cache: Dict[str, tuple] = {}
        self._ttl = ttl
    
    def get(self, key: str) -> Optional[Any]:
        if key in self._cache:
            value, expiry = self._cache[key]
            if time.time() < expiry:
                return value
        return None
    
    def set(self, key: str, value: Any) -> None:
        self._cache[key] = (value, time.time() + self._ttl)

6.3 SPARQL Query Examples

6.3.1 DBpedia: Fetching Artwork Abstract

PREFIX dbo: <http://dbpedia.org/ontology/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

SELECT ?abstract ?thumbnail ?artistName ?museumName WHERE {
    <http://dbpedia.org/resource/Mona_Lisa> dbo:abstract ?abstract .
    FILTER(LANG(?abstract) = "en")
    
    OPTIONAL { ?artwork dbo:thumbnail ?thumbnail }
    OPTIONAL { 
        ?artwork dbo:author ?artist .
        ?artist rdfs:label ?artistName .
        FILTER(LANG(?artistName) = "en")
    }
    OPTIONAL {
        ?artwork dbo:museum ?museum .
        ?museum rdfs:label ?museumName .
        FILTER(LANG(?museumName) = "en")
    }
}

6.3.2 Wikidata: Fetching Artwork Metadata

PREFIX wd: <http://www.wikidata.org/entity/>
PREFIX wdt: <http://www.wikidata.org/prop/direct/>
PREFIX wikibase: <http://wikiba.se/ontology#>
PREFIX bd: <http://www.bigdata.com/rdf#>

SELECT ?image ?inception ?creatorLabel ?locationLabel ?materialLabel WHERE {
    OPTIONAL { wd:Q12418 wdt:P18 ?image }
    OPTIONAL { wd:Q12418 wdt:P571 ?inception }
    OPTIONAL { wd:Q12418 wdt:P170 ?creator }
    OPTIONAL { wd:Q12418 wdt:P276 ?location }
    OPTIONAL { wd:Q12418 wdt:P186 ?material }
    
    SERVICE wikibase:label { bd:serviceParam wikibase:language "en" }
}

6.3.3 Getty AAT: Material Term Lookup

PREFIX skos: <http://www.w3.org/2004/02/skos/core#>
PREFIX gvp: <http://vocab.getty.edu/ontology#>

SELECT ?prefLabel ?scopeNote ?broaderLabel WHERE {
    <http://vocab.getty.edu/aat/300015050> skos:prefLabel ?prefLabel .
    FILTER(LANG(?prefLabel) = "en")
    
    OPTIONAL { 
        <http://vocab.getty.edu/aat/300015050> skos:scopeNote [rdf:value ?scopeNote] .
        FILTER(LANG(?scopeNote) = "en")
    }
    OPTIONAL {
        <http://vocab.getty.edu/aat/300015050> gvp:broaderGeneric ?broader .
        ?broader skos:prefLabel ?broaderLabel .
        FILTER(LANG(?broaderLabel) = "en")
    }
}

6.3.4 Federated Query: Cross-Repository Enrichment

PREFIX arp: <http://example.org/arp#>
PREFIX owl: <http://www.w3.org/2002/07/owl#>
PREFIX dbo: <http://dbpedia.org/ontology/>

SELECT ?artwork ?title ?abstract WHERE {
    ?artwork a arp:Artwork ;
             dc:title ?title ;
             owl:sameAs ?dbpediaUri .
    
    SERVICE <https://dbpedia.org/sparql> {
        ?dbpediaUri dbo:abstract ?abstract .
        FILTER(LANG(?abstract) = "en")
    }
}

6.4 Error Handling & Timeouts

External queries are wrapped with timeout handling (10 seconds) and graceful degradation. If an external source is unavailable, the API returns local data without enrichment rather than failing entirely.

7. Linked Data Conformance

7.1 The Four Linked Data Principles

ArP adheres to Tim Berners-Lee's four principles for Linked Data:

Principle 1: Use URIs as Names for Things

Every entity in ArP has a globally unique URI. Artworks use the pattern http://example.org/arp#artwork_{id}, artists use http://example.org/arp#artist_{id}, and provenance events use http://example.org/arp#provenance_{id}.

Principle 2: Use HTTP URIs

All URIs use the HTTP scheme, making them potentially dereferenceable. When deployed with a proper domain, these URIs could resolve to HTML or RDF representations via content negotiation.

Principle 3: Provide Useful Information Using Standards

When a URI is accessed (via the API or SPARQL), the system returns data in standard formats:

  • JSON with embedded JSON-LD context for web applications
  • Native RDF (Turtle, N-Triples, RDF/XML) via the Fuseki SPARQL endpoint
  • Standard vocabularies (Dublin Core, Schema.org, PROV-O, CIDOC-CRM)

Principle 4: Include Links to Other URIs

ArP actively links to external Linked Open Data resources:

  • owl:sameAs links to DBpedia resources (e.g., dbr:Mona_Lisa)
  • owl:sameAs links to Wikidata entities (e.g., wd:Q12418)
  • schema:artMedium links to Getty AAT terms (e.g., aat:300015050)

7.2 Five-Star Open Data

ArP achieves a 5-star rating on Tim Berners-Lee's 5-star deployment scheme:

Stars Requirement ArP Status
★ Data available on web with open license ✓ MIT License
★★ Machine-readable structured data ✓ JSON, RDF
★★★ Non-proprietary format ✓ RDF/Turtle, JSON-LD
★★★★ Use URIs to identify things ✓ HTTP URIs throughout
★★★★★ Link to other data for context ✓ DBpedia, Wikidata, Getty

7.3 FAIR Principles Alignment

The platform also aligns with FAIR principles for research data:

  • Findable: Unique identifiers, searchable metadata, indexed in triple store
  • Accessible: Open REST API, standard SPARQL endpoint, no authentication for public data
  • Interoperable: Standard vocabularies, formal ontology, machine-readable formats
  • Reusable: Clear licensing, provenance tracking, vocabulary documentation
